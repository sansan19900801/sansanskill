# `sansan-restore`｜三三商业诊断恢复

从 `~/.sansan/records/` 找回上次诊断状态，先核对新变化，再决定继续旧动作还是重新诊断。

**只负责恢复历史状态，不把旧结论自动当成当前事实。**

## 核心边界

1. 用户主动要求恢复，即视为授权读取对应项目的本地存档。
2. 只读，不修改、覆盖或重新保存原存档。
3. 恢复内容必须标明“这是当时的诊断状态”。
4. 当前对话已经提供的新事实优先于旧存档。
5. 不自动执行旧 `next_skill`；必须先确认变化或得到用户明确指令。
6. 不读取用户未选择的其他项目正文，不上传任何记录。

## 模式

| 输入 | 行为 |
| --- | --- |
| `/sansan-restore`、接着上次 | 恢复当前项目最新存档 |
| `/sansan-restore 2` | 恢复列表中的第 2 份存档 |
| `/sansan-restore list` | 列出当前项目存档 |
| 指定项目并恢复 | 只读取指定项目 |

## 恢复流程

### 1. 确定项目

优先使用用户明确指定的项目；其次使用当前对话已经确认的业务。无法判断时只问：

> 你要接着哪个业务或项目的诊断？

不要根据当前工作目录盲猜，也不要默认混入 `default`。

### 2. 定位存档

优先调用 `sansan-save` 随附的 `record_store.py`：

```text
python3 record_store.py resolve --project <项目> --selector <latest或序号>
```

脚本从已安装的 `sansan-save/scripts/` 真实路径解析。找不到该脚本时，可直接只读 `~/.sansan/records/{项目}/`，按文件名前缀时间排序，不使用 mtime。

当前项目没有存档时，可用 `projects` 命令只列出最近 3 个项目名称和日期；未经用户选择，不读取这些项目正文。

### 3. 容错读取

读取完整 Markdown 和 frontmatter。字段缺失时尽量展示已有内容，同时标注“字段缺失”；不得因此编造。`schema_version` 高于当前可识别版本时，提示版本不兼容，不擅自改写。

### 4. 紧凑呈现

完整读取 [restore-view.md](references/restore-view.md)，只展示关键状态，不全文复述。

### 5. 核对变化

先复用当前对话里已经出现的新事实。没有新结果时只问：

> 上次之后，实际发生了什么变化？如果没有变化，可以直接说“没有变化，按上次继续”。

- 有新结果：把旧状态和新事实一并交给 `sansan-business-diagnosis` 复检。
- 没有变化且用户明确按上次继续：重新检查 `next_skill` 是否存在并标记为 `ready`，再把存档核心内容作为上下文交过去。
- 用户换了问题：按 `/sansan` 当前路由重新判断，不强迫继承旧结论。

## list 模式

按最新到最早展示日期、标题、状态和来源 Skill。序号必须与 `resolve` 模式一致。用户指定超出范围的序号时报告真实数量。

## 边界情况

- 文件被删除：说明已不存在，并建议查看列表；
- 文件被手动修改：尽量读取，明确缺失字段；
- 连续恢复同一份：提示已在当前对话展示，除非用户要换一份；
- `next_skill` 已下线：不调用，回到 `/sansan` 重新路由；
- 当前事实与旧记录冲突：标记冲突并进入复检，不替用户选择。

## 输出自检

- 项目是否由用户或当前上下文确认；
- 是否误读其他项目正文；
- 是否把历史状态写成当前事实；
- 是否优先使用当前新证据；
- 是否修改了原存档；
- 是否在未经确认时自动执行旧动作。
