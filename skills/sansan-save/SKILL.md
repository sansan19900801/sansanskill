# `sansan-save`｜三三商业诊断存档

把当前对话中已经确认的商业事实、诊断结论、被排除方向、待验证假设和下一步动作保存为本地 Markdown 存档。

**只负责保存，不重新诊断。**

## 核心边界

1. 只有用户明确要求保存时才能执行；“保存这次诊断”即视为授权保存本次诊断中的必要经营数据，不再重复询问。
2. 不自动保存任何对话或商业资料。
3. 只把用户确认过的事实写进“已确认经营事实”；推测必须放进“待验证假设”。
4. 客户联系方式、身份证、付款账号等与诊断无关的高敏感信息默认脱敏。
5. 存档只写入用户本机 `~/.sansan/records/`，不得上传 GitHub、Obsidian 或外部服务。
6. 没有本地文件权限时只生成待保存正文，并明确说明没有真正写入。

## 模式

| 输入 | 行为 |
| --- | --- |
| `/sansan-save`、保存这次诊断 | 保存当前已确认状态 |
| `/sansan-save 标题` | 使用指定标题保存 |
| `/sansan-save list` | 列出当前项目的存档，不写新文件 |

## 保存流程

### 1. 检查是否可保存

必须已经形成真实诊断结果。只有一句需求、尚未完成诊断或没有任何确认结论时，回复：

> 当前还没有形成可保存的商业诊断。先完成一次商业问诊或商业体检，再保存。

不得生成空存档。

### 2. 确定项目与标题

- 优先使用用户明确指定的项目；其次使用当前正在诊断的产品或业务。
- 无法判断时只问：“这次诊断要归到哪个业务或项目？”
- 禁止把无法判断的记录混入 `default`。
- 标题不超过 20 个汉字，直接描述问题或结论，不包含客户隐私。

### 3. 生成存档正文

完整读取 [record-schema.md](references/record-schema.md)，严格区分事实、判断、假设和行动。没有信息的字段写“暂无”或省略，不编造。

### 4. 写入本地

优先使用 `scripts/record_store.py` 的 `write` 命令原子写入：

```text
python3 scripts/record_store.py write --kind record --project <项目> --title <标题> --content-file <正文文件>
```

脚本必须从本 Skill 的真实安装目录解析，不假定当前工作目录。写入后读取脚本返回的 JSON，确认 `path` 存在。

如果不能运行脚本，但具备本地文件能力，可按同一目录和命名规则写入；必须新建文件、不得覆盖历史。没有任何文件能力时只输出正文。

### 5. 回执

真实写入后只回复：

> 已存档：`{路径}`
>
> 当前项目共 {N} 份存档。下次可以说“接着上次继续”。

不要复述全文，不说“保存成功”。

## list 模式

调用脚本：

```text
python3 scripts/record_store.py list --kind record --project <项目>
```

按最新到最早展示：日期、标题、状态和来源 Skill。状态翻译为“进行中 / 已结论 / 已放弃”。没有记录时直接说明。

## 状态

- `in-progress`：诊断或验证仍在进行；
- `resolved`：已经形成稳定结论；
- `abandoned`：用户明确放弃该方向。

不确定时使用 `in-progress`。

## 输出自检

- 用户是否明确要求保存；
- 是否存在真实诊断结果；
- 事实、判断、假设和建议是否分开；
- 是否写入未经确认的结论；
- 是否包含不必要的高敏感信息；
- 项目是否归类正确；
- 文件是否真实存在；
- 是否误传到仓库或外部服务。
