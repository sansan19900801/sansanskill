# `sansan-report`｜三三商业诊断报告

把 `~/.sansan/records/` 中已经确认的诊断状态整理成单次诊断报告或阶段经营报告。

**报告只重组已有存档，不重新诊断，不修改原始记录。**

## 报告类型

- 1 份存档：生成“单次商业诊断报告”；
- 2 份及以上：生成“阶段商业经营报告”；
- 用户说“给客户、学员、合伙人看”或“对外分享”：生成脱敏分享版；
- 其他情况默认生成内部完整版。

## 核心边界

1. 只读取用户选定项目 `~/.sansan/records/` 下的正式存档。
2. 不从零散聊天补充未存档结论。
3. 不修改、删除或覆盖任何原始存档。
4. 每次生成新报告，写入 `~/.sansan/reports/`。
5. 每个重要结论必须能追溯到存档标题和日期。
6. 新事实可以修正旧判断，但旧判断仍保留并标注修正关系。
7. 没有执行结果时写“尚未回流”，不能编造成果。

## 生成流程

### 1. 确定项目、范围和受众

优先复用用户已经指定的项目。无法判断时只问项目。用户未指定时间范围时读取全部正式存档；明确说“最近30天”或指定日期时再过滤。

受众规则：

- 默认：内部完整版；
- 明确给客户、学员、合伙人或对外使用：脱敏分享版。

### 2. 读取存档

按文件名前缀时间从早到晚读取。格式缺失时尽量使用现有字段并标注，不因为单个坏文件中断全部报告。

没有存档时停止；一份存档也允许生成报告，不要求用户强制确认。

### 3. 合并规则

- 事实：最新有证据记录优先，旧事实保留在演进部分；
- 结论：语义相近可合并，但必须保留来源；
- 冲突：同时呈现，不擅自裁决；
- 否决方向：保留方向、原因、日期；
- 待验证假设：只收录最新存档明确标为待验证的内容；
- 动作：区分计划、已执行、已回流和结果未知；
- `next_skill`：只作历史建议，不能写成当前已决定动作。

### 4. 生成正文

完整读取 [report-schema.md](references/report-schema.md)，不得删除来源索引。

分享版必须默认脱敏：客户姓名、联系方式、身份证、付款账号、未公开合同信息和不必要的内部经营隐私。经营数字只有在用户明确要求保留时才进入分享版。

### 5. 写入本地

优先使用 `sansan-save/scripts/record_store.py`：

```text
python3 record_store.py write --kind report --project <项目> --title <报告标题> --content-file <报告正文文件>
```

脚本从已安装的 `sansan-save` 真实路径解析。没有脚本但具备本地文件能力时，按相同规则新建报告；不得覆盖旧文件。没有文件权限时只输出报告正文并说明未写入。

### 6. 回执

真实写入后只回复：

> 报告已生成：`{路径}`
>
> 使用了 {N} 份存档（{起始日期} → {结束日期}），版本：{内部完整版/脱敏分享版}。

## 输出自检

- 报告是否只来自正式存档；
- 重要结论是否可追溯；
- 是否把旧状态误写成当前事实；
- 是否掩盖了冲突或修正；
- 是否编造执行结果；
- 分享版是否完成脱敏；
- 是否覆盖原始存档或旧报告；
- 文件是否真实存在。
