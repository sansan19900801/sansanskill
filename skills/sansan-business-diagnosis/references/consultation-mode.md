# 商业问诊模式

## 目的

处理一个具体商业问题。先判断问题是否成立、是否值得现在回答，再给结论。不要为了展示流程强行走完所有关卡。

## 第一步：复述决策任务

用一句话复述：

> 你现在真正要决定的是【决策】，希望影响【经营结果】，当前依据是【已知事实】。

如果用户只是表达感受，追问一个问题：

> 这次聊完，你具体要做出什么决定？

## 第二步：问题有效性检查

按顺序检查，某一步已经足以重写或消解问题时先停下来与用户确认。

### 1. 结果锚点

检查“好、适合、值得、有效、高端、有流量”等词具体指什么可观察结果。不是要求所有结果都量化，而是要能判断有没有发生。

输出：原词、可能含义、本次采用的可检查定义。

### 2. 隐藏前提

把问题改写成：

> 这个问题假设了【前提】，目前有什么事实证明它成立？

常见前提：缺流量、价格太高、客户没钱、需要更多AI工具、内容多就会成交、产品多就能形成矩阵。

### 3. 因果链

检查用户是否把先后发生、相关或成功者特征当成因果。写出：

`观察事实 → 用户解释 → 可能的其他解释 → 还缺什么证据`

### 4. 事实底座

区分真实付款、咨询、发布、交付、复购等行为，与“感觉、应该、别人都说、市场趋势”之间的差别。关键事实未核实时，不给精确答案，给最小取证动作。

### 5. 决策条件

判断属于哪一类：

- `可直接判断`：事实与标准足够；
- `需要最小实验`：缺市场行为证据；
- `价值选择`：没有唯一正确答案；
- `资源约束`：结论取决于时间、现金、能力或团队；
- `超出边界`：法务、财税、医疗等需专业人士。

## 第三步：回答或重写问题

### 问题被重写

输出：

1. 原问题为什么容易把人带错；
2. 更准确的问题；
3. 现在能判断什么；
4. 还不能判断什么。

### 问题可以回答

输出：

1. 明确结论；
2. 支撑事实；
3. 因果链；
4. 反方解释；
5. 最大风险；
6. 最小验证动作；
7. 推翻条件。

## 结束回顾

简短回顾：

> 你最初问的是【原问题】。经过检查，真正需要处理的是【真问题】。当前结论是【结论】，下一步只做【动作】。如果出现【推翻条件】，我们重新判断。
