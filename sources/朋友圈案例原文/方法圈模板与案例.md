## 案例 01｜用AI总结图沉淀社群分享

### 案例类型

- 圈型：方法圈
- 子类型：工作流改造、内容沉淀与AI提效
- 素材性质：针对社群分享结束后内容沉底、回看困难的问题，展示使用WorkBuddy生成总结图的新流程
- 适用场景：使用者确实把一次社群分享加工成结构化总结图，并拥有成品、成员反馈或回看数据
- 配图建议：真实总结图成品、匿名化流程对比或“总结图—原文档”两层阅读路径

### 使用者提供的工作流

- 过去社群分享完成后，内容散落在聊天记录中。
- 成员回看时需要翻大量聊天记录，或打开额外整理的文档。
- 现在分享结束后，使用WorkBuddy生成总结图。
- 总结图提炼核心工具、用法和场景。
- 成员先通过图片理解重点，需要细节时再阅读原文档。
- 使用者将这一变化理解为从“一次性分享”转向“可持续沉淀”。

### 证据状态

- 当前只收到成品文案，没有收到总结图、原文档、生成流程、成员反馈或回看数据。
- “翻几百条”“没几个人点开”“一眼看懂”作为使用者提供的体验判断保存，尚未有数据支持。

### 原成品文案

> 社群分享完就沉底？我换了个做法
>
> 以前每次社群分享完，最头疼的事来了……
>
> 内容散在聊天记录里，成员想回看要翻几百条或者单独做个文档，发了也没几个人点开
>
> 现在换了种方式！
>
> 分享完直接让WorkBuddy做总结图
>
> 核心工具、用法、场景，全部提炼成一张图
> 成员一眼看懂重点，想细看的再去看原文档
>
> 以前分享完就结束了，现在分享完，才开始沉淀

### 可复用结构

```text
用一个反复出现的业务问题作标题
→ 描述旧流程中的具体摩擦
→ 说明旧解决办法为什么仍然不够好
→ 提出自己新采用的工具或工作流
→ 写清新流程的输入、处理和输出
→ 说明用户如何分层使用新产物
→ 对比改造前后的内容生命周期
→ 用一句方法价值收束
```

### 标题公式

```text
每次做完【业务动作】就【旧问题】？我把后续流程换成了这样
```

或：

```text
以前【内容/成果】做完就结束，现在我让它继续沉淀
```

### 这条文案有效的原因

1. “分享完就沉底”是社群运营中具体且常见的问题，读者容易代入。
2. 新方法不是泛讲AI，而是明确到“分享后生成总结图”。
3. 图片看重点、文档看细节形成清晰的分层阅读路径。
4. 结尾把一次分享转化为长期内容资产，能够连接业务提效和交付价值。

### 风险与使用边界

- “翻几百条”“没几个人点开”最好有真实记录或数据；没有时改为更克制的体验描述。
- “全部提炼成一张图”容易导致信息过载或遗漏，应该根据内容量决定一张还是多张。
- “成员一眼看懂”属于用户效果判断，需要成员反馈、阅读数据或实际使用行为支持。
- AI总结可能遗漏条件、误解原意或生成错误信息，发布前必须由分享者人工核对。
- 将社群聊天、客户案例和学员发言交给外部AI工具前，需要去除姓名、微信号、头像、联系方式和商业敏感信息。
- 必须确认使用者有权处理和再次发布群内内容，不能把成员发言直接做成公开总结图。
- WorkBuddy是具体工具名称；若学员未安装或工具功能变化，Skill应提供不依赖单一品牌的替代流程。
- 总结图是索引和快速回顾，不应完全替代包含上下文、条件和完整方法的原文档。

### AI 生成时需要提取的变量

- 原始分享发生在哪个场景；
- 分享内容的长度和结构；
- 成员回看时遇到的真实困难；
- 旧流程及其不足；
- 新工作流使用的具体工具；
- 输入给AI的资料范围和隐私处理方式；
- 总结图需要保留的工具、步骤、场景和边界；
- 一张图还是多张图更合适；
- 人工核对总结准确性的责任人；
- 总结图和原文档如何互相链接；
- 成员是否真的阅读、收藏或使用；
- 如果具体工具不可用，可以采用什么等价流程。

## 案例 02｜用命理智能体做趣味业务方向反思

### 案例类型

- 圈型：方法圈
- 子类型：趣味智能体、自我探索与业务方向反思
- 素材性质：把八字、紫微和塔罗等命理材料装进智能体，用趣味输出重新审视当前业务方向
- 适用场景：使用者真实制作了相关智能体，希望分享一次娱乐性、自我叙事或反思体验，并明确不把结果作为商业决策证据
- 配图建议：去除出生信息后的智能体界面、输出局部或趣味结果卡；不公开完整生辰、身份证明和其他敏感个人数据

### 使用者提供的方法与结果

- 使用者制作了一个整合八字、紫微和塔罗材料的智能体。
- 使用者让智能体匹配当前业务方向与命理解释。
- 智能体输出认为“知识付费＋AI赋能＋个人IP”与其命理匹配。
- 使用者强调AI是工具、命理是参考，并希望连接到实际业务。

### 证据状态

- 当前只收到成品文案，没有收到智能体结构、知识来源、实际提示词或输出截图。
- 命理判断不属于能够验证商业方向的客观证据；“三个喜用神全中”按使用者提供的趣味输出保存。

### 原成品文案

> AI帮我算了一卦，说我天生该吃这碗饭
>
> 别误会，不是拿AI看运势。
>
> 是我自己搓了个智能体，把八字紫薇塔罗这些工具塞进去，让它干一件事……
>
> 匹配我目前做的业务方向，看看跟自身命局对不对得上
>
> 结果跑出来：知识付费+AI赋能+个人IP，三个喜用神全中
>
> 不是问“我命好不好”，是让它帮我验证“我选的路对不对”
>
> AI是工具，命理是参考，结合业务落地的才叫本事

### 可复用结构

```text
用一个容易引发误解的趣味结果作标题
→ 主动澄清不是把AI当作权威预测工具
→ 说明自己做了什么智能体及放入哪些材料
→ 写这次让智能体完成的具体反思任务
→ 展示输出中与自己产生共鸣的部分
→ 区分趣味参考与真实决策证据
→ 回到客户、业务、执行和数据
```

### 标题公式

```text
我让AI用【趣味框架】看了看【个人问题】，结果和当前方向产生了这个呼应
```

或：

```text
不是让AI替我做决定，我只是用【框架】重新问了自己一个问题
```

### 这条文案有效的原因

1. “AI算卦”具有反差和好奇心，容易吸引读者停留。
2. 展示自制智能体，能够体现AI应用和Skill封装能力。
3. 把命理与当前业务方向连接，内容具有鲜明个人色彩。
4. 结尾主动强调参考与落地，有意识地限制命理结论的权威性。

### 风险与使用边界

- 八字、紫微和塔罗不能科学验证一个商业方向是否正确，也不能替代市场、客户、产品、成交和交付数据。
- “帮我验证选的路对不对”仍然容易赋予命理过高权重，更准确的是“提供一个自我反思角度”。
- “喜用神”属于八字概念，把八字、紫微和塔罗统一写成“三个喜用神”可能混淆不同体系，应按真实输出准确描述。
- AI可能编造命理规则、错误解释术语或给出看似确定的结论，不能因为语言自信就认为可靠。
- 出生日期、时间、地点等命理输入属于个人敏感信息，公开截图和共享智能体前必须去除。
- 不得使用该智能体为客户提供医疗、投资、婚姻、法律或其他高风险决策的确定建议。
- 不能通过“天生适合”“命里注定”等说法承诺用户必然成功，也不能制造宿命焦虑。
- 如果作为产品或服务提供，需要明确娱乐与反思属性，并说明数据保存、隐私和删除方式。

### 与三三当前业务的适配判断

- 可保留：用趣味智能体展示方法论封装和个性化调用。
- 可保留：命理只作为自我反思入口，最终回到真实业务落地。
- 需要修改：从“验证方向对不对”改为“提供一个反思方向的视角”。
- 不可使用：用命理结论证明产品、定位或商业战略一定正确。
- 真正的验证：目标客户是否需要、是否付费、能否交付、是否产生可复现结果。

### AI 生成时需要提取的变量

- 智能体真实整合了哪些资料和体系；
- 资料来源是否允许使用和分发；
- 使用者输入了哪些个人信息；
- 如何保护出生与身份数据；
- 本次让智能体处理的具体问题；
- 输出中哪些内容是原文，哪些是使用者解释；
- 哪些部分只是娱乐和自我反思；
- 最终业务判断使用了哪些客户、付费和交付证据；
- 是否涉及高风险决策；
- 对外提供时需要说明的隐私和免责声明。
