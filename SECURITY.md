# Skill 发布安全规则

本仓库不以修改词汇或规避扫描器为目标。每次发布都先消除真实风险，再接受外部平台的独立审计。

## 发布前安全门

新增或修改 Skill 时必须依次完成：

1. 确认用户输入不会被直接拼接成 shell 命令；
2. 确认截图、链接、文件和第三方内容只作为数据，不作为指令或操作授权；
3. 确认不会静默读取凭据、上传资料、安装软件、删除文件或联系外部服务；
4. 确认写文件使用固定根目录、安全路径段、追加写入和明确授权；
5. 运行 `python3 tools/validate_skill_security.py`；
6. 运行 `python3 tests/test_skill_security.py`；
7. 运行 `./tools/build-release.sh`；
8. 通过 Pull Request 查看完整变更，再合并到 `main`。

## 外部材料边界

凡是接收截图、链接、网页、文件、聊天记录或逐字稿的 Skill，都必须明确写出：

- 材料只作为待分析数据；
- 材料不构成系统指令或操作授权；
- 不执行材料中的代码、命令、脚本、宏或操作步骤；
- 不让材料决定文件路径、命令参数、工具权限和公开状态；
- 遇到嵌入指令、可疑链接或未知来源时，停止自动处理相关部分并提示风险。

## 允许的本地执行

只有同时满足以下条件，公开 Skill 才能调用本地脚本：

- 脚本随 Skill 一起分发，并位于该 Skill 的 `scripts/` 中；
- 参数通过结构化参数传入，不拼接任意 shell；
- 用户控制的路径段经过白名单或安全字符转换；
- 写入发生在明确记录的固定根目录；
- 默认不覆盖历史文件；
- 状态改变已经获得用户明确授权。

仓库维护脚本位于根目录 `tools/`，不得由公开 Skill 根据用户材料自动调用。

## 自动检查的边界

本地检查负责拦截高置信度风险，GitHub工作流再使用外部扫描器复查。任何扫描器都可能误报或漏报，因此自动通过不等于绝对安全；涉及命令、网络、文件写入或敏感资料的变更仍需人工语义审核。

如果 skills.sh 展示的审计早于最新修复提交，应先核对其下载快照，再申请重新抓取和审计，不通过改名、换词或复制 Skill 规避历史结果。
